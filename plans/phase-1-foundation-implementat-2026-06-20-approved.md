# Phase 1 Foundation Implementation Plan

## Scope Boundary

Phase 1 should establish the service foundation only: TypeScript scaffold, Fastify bootstrap, health endpoint, typed env config, structured logging, Vitest + `app.inject()` test setup, lint/format/typecheck/build scripts, CI, Docker Compose Postgres, and initial README sections.

Do not implement product domain behavior, Drizzle tables, migrations for product data, product routes, search, pagination, or repositories beyond minimal DB connection scaffolding if needed for config validation. Swagger/OpenAPI can wait unless Phase 1 scope is explicitly expanded; when added later, use `@fastify/swagger` and `@fastify/swagger-ui` separately and register Swagger before routes.

## Execution Principles

Use TDD at each logical unit: add or outline the failing test first, implement the smallest code path to pass, then run the narrow check and the relevant full check.

Keep the architecture layer-first from the first commit:

```text
src/
  app/
  routes/
  schemas/
  services/
  repositories/
  db/
  errors/
  ids/
test/
  helpers/
  integration/
  unit/
```

For Phase 1, several folders may contain only focused foundation files or remain absent until used. Tests must live under repo-root `test/`, not under `src/`.

Use ESM-first TypeScript. Fastify 5 should target Node 20 or 22. TypeBox 1.x is ESM-only, so avoid CommonJS config patterns. Use `typebox` plus `@fastify/type-provider-typebox`.

## Recommended Atomic Commit Strategy

Commit 1: Project Toolchain Baseline

Purpose: Create the Node/TypeScript/pnpm project foundation and runnable quality gates.

Implement:

- `package.json` with `type: "module"`, pnpm package manager metadata, Node engine targeting Node 20/22, and scripts for `format`, `format:check`, `lint`, `typecheck`, `test`, `test:watch`, `build`, and optionally `start`/`dev`.
- `pnpm-lock.yaml` generated by install.
- TypeScript config for ESM Node, strict mode, source maps, declaration output if useful, and clean `dist/` output.
- ESLint flat config suitable for TypeScript and ESM.
- Prettier config.
- Vitest config with Node environment and tests rooted under `test/`.
- Minimal `.env.example` only if env names are already known.
- Update `.gitignore` for `node_modules`, `dist`, coverage, local env files, and tooling outputs.

TDD angle:

- No app behavior yet. Treat the quality gates themselves as the first executable contract.
- Keep this commit focused on making empty or minimal checks run deterministically.

Validation after this unit:

```bash
pnpm install
pnpm format:check
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

Risk points:

- ESM config friction across TypeScript, Vitest, ESLint, and Prettier.
- TypeBox 1.x ESM-only means any CommonJS assumptions will fail later.
- Keep dependency list lean; do not introduce alternative HTTP, ORM, schema, DB, or test frameworks.

Commit 2: App Bootstrap And Health Contract

Purpose: Add the smallest Fastify app with a health route and inject-based integration test.

Implement tests first:

- Add `test/integration/health.test.ts` using a test helper that builds the app and closes it in teardown.
- Assert `GET /health` returns `200`, JSON content, and a stable response shape such as `{ "status": "ok" }`.
- If choosing to include uptime/timestamp later, avoid flaky exact-value assertions.

Implement app code:

- `src/app/build-app.ts` creates and returns a Fastify instance with `.withTypeProvider<TypeBoxTypeProvider>()`.
- Use Fastify 5 with `logger: true` initially or a logger option derived from config in the next commit.
- `src/routes/health.routes.ts` registers only route metadata/schema and delegates response creation if any logic exists.
- `src/schemas/health.schemas.ts` holds the TypeBox response schema.
- `src/app/server.ts` or `src/main.ts` starts the server only for runtime, not tests.
- Ensure tests use `app.inject()` and close the app in teardown.

Validation after this unit:

```bash
pnpm test -- test/integration/health.test.ts
pnpm lint
pnpm typecheck
pnpm build
```

Risk points:

- Avoid listening on a real port during tests.
- Avoid route files becoming mini-apps; they should only register endpoints, attach TypeBox schemas, read input, call services if needed, and return responses.
- Keep the health endpoint intentionally simple; no database dependency in Phase 1 health unless explicitly required.

Commit 3: Typed Runtime Config

Purpose: Centralize and validate environment configuration at startup.

Implement tests first:

- Add unit tests under `test/unit/config.test.ts` for defaults, valid overrides, invalid port, invalid log level, and missing required values if any.
- Tests should call a pure config-loading function with an injected env object, not mutate global process state when avoidable.

Implement:

- `src/app/config.ts` or `src/app/env.ts` loads and validates env once.
- Include practical Phase 1 values: `NODE_ENV`, `HOST`, `PORT`, `LOG_LEVEL`, and `DATABASE_URL` if Docker Compose/Postgres is part of local setup.
- Use a typed runtime validation approach compatible with the stack. If adding a small dependency, justify it; otherwise implement a small explicit parser.
- Runtime server entrypoint reads config and passes it into app creation.
- Tests should not require real env files.

Validation after this unit:

```bash
pnpm test -- test/unit/config.test.ts
pnpm test
pnpm lint
pnpm typecheck
pnpm build
```

Risk points:

- Do not read `process.env` throughout the codebase.
- Do not make test order depend on global env mutation.
- Decide whether `DATABASE_URL` is required at server startup in Phase 1. If the app does not connect to the DB yet, it may be better to validate it only for DB tasks or document it as required for future DB-backed behavior.

Commit 4: Structured Logging Baseline

Purpose: Make logging production-appropriate while keeping tests quiet and deterministic.

Implement tests first:

- Add focused unit tests for logger option selection by environment, if this logic is non-trivial.
- Existing health inject tests should still pass with logging enabled or disabled depending on test config.

Implement:

- Keep Fastify’s built-in Pino logger.
- Use JSON structured logs outside local development.
- Use pretty logging only in local development if `pino-pretty` is added; otherwise keep `logger: true` and document plain JSON logs.
- Disable or minimize logs in tests if needed.
- Redact secrets if request or config logging is introduced.
- Be careful with `requestIdHeader`; user-supplied request IDs can be spoofed, so do not rely on them for trust/security decisions.

Validation after this unit:

```bash
pnpm test
pnpm lint
pnpm typecheck
pnpm build
```

Risk points:

- Avoid logging config secrets or `DATABASE_URL`.
- Avoid over-customizing Pino before operational needs exist.
- If `pino-pretty` is added, keep it dev-only.

Commit 5: Problem Details Error Baseline

Purpose: Establish safe, consistent error responses before future routes exist.

Implement tests first:

- Add an integration test that registers a test-only route or uses an internal app option to trigger a known validation/error path, if practical.
- Assert unexpected errors become safe Problem Details-style 500 responses without leaking internals.
- Assert validation failures become consistent 400 responses if Fastify validation is wired at this point.

Implement:

- `src/errors/problem-details.ts` for the RFC 9457-style response shape.
- `src/errors/error-handler.ts` maps Fastify validation errors and unexpected errors to safe JSON responses.
- Register the error handler in `build-app`.
- Keep implementation minimal; do not create a large app-error hierarchy until product behavior exists.

Validation after this unit:

```bash
pnpm test
pnpm lint
pnpm typecheck
pnpm build
```

Risk points:

- Do not return raw internal errors to clients.
- Keep route-specific business errors out of Phase 1 unless needed by health/config tests.
- Ensure response schemas and actual responses stay aligned.

Commit 6: Postgres Local Development Compose

Purpose: Provide repeatable local Postgres for later DB-backed phases.

Implement:

- `docker-compose.yml` with a Postgres service, named volume, environment variables suitable for local development, published local port, and a `pg_isready` healthcheck.
- `.env.example` with local database connection values if not already added.
- Optional `scripts` entries such as `db:up`, `db:down`, and `db:logs` if they improve repeatability.
- Do not add product schema migrations yet.

Validation after this unit:

```bash
docker compose config

docker compose up -d postgres

docker compose ps

docker compose down

pnpm format:check
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

Risk points:

- Short-form `depends_on` does not wait for Postgres readiness. Use a healthcheck for readiness where dependencies are introduced.
- Do not require Docker for the health endpoint tests unless a DB-backed test is introduced later.
- Avoid committing local `.env` secrets.

Commit 7: CI Workflow

Purpose: Make every Phase 1 quality gate run on pull requests and pushes.

Implement:

- GitHub Actions workflow under `.github/workflows/ci.yml`.
- Use Node 20 or 22 consistently with `package.json` engines.
- Enable pnpm via Corepack or pnpm setup action.
- Install with frozen lockfile.
- Run `format:check`, `lint`, `typecheck`, `test`, and `build`.
- Include a Postgres service only if tests or future smoke checks need it. If no DB tests exist yet, keep CI simpler and let Docker Compose be local-dev only for Phase 1.

Validation after this unit locally:

```bash
pnpm format:check
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

Optional local workflow validation if available:

```bash
act -j ci
```

Risk points:

- CI must use pnpm, not npm.
- Frozen lockfile will fail if `pnpm-lock.yaml` is stale.
- If adding a CI Postgres service, align credentials with `.env.example` and avoid requiring it for non-DB tests.

Commit 8: Initial README Foundation

Purpose: Document the actual Phase 1 workflow after scripts and files exist.

Implement near the end, not first:

- Project overview and current Phase 1 status.
- Requirements: Node 20/22, pnpm, Docker for local Postgres.
- Setup commands: install, env copy, optional Docker Compose up, run tests, run dev server.
- Quality commands: format, lint, typecheck, test, build.
- Architecture summary: layer-first layout and tests under `test/`.
- Environment variables table.
- Health endpoint example.
- Scope notes: product routes, Drizzle migrations, Swagger/OpenAPI, pagination, and search come in later phases.

Validation after this unit:

```bash
pnpm format:check
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

Risk points:

- Do not document commands that do not exist.
- Do not imply product API behavior is available in Phase 1.
- Keep README aligned with the scripts and compose file actually implemented.

## Ultrawork Execution Order

Recommended implementation order:

1. Toolchain baseline.
2. Health test contract.
3. Fastify app factory and health route.
4. Typed config.
5. Logging baseline.
6. Error handling baseline.
7. Docker Compose Postgres.
8. CI workflow.
9. README updates.
10. Final full validation pass.

What to implement first:

- `package.json`, TypeScript, lint/format/test/build tooling, and a minimal test runner setup.
- The first actual app behavior should be the `GET /health` inject test, followed by app bootstrap implementation.

What can wait until the end:

- README content that depends on final script names.
- CI workflow polish after local scripts are stable.
- Docker Compose script aliases if the basic Compose file works.
- Swagger/OpenAPI; defer unless Phase 1 scope is expanded.
- Drizzle schema and migrations; defer to the first DB-backed phase.

## Final Phase 1 Validation Gate

Before declaring Phase 1 complete, run:

```bash
pnpm install --frozen-lockfile
pnpm format:check
pnpm lint
pnpm typecheck
pnpm test
pnpm build
docker compose config
docker compose up -d postgres
docker compose ps
docker compose down
```

If CI is already pushed/opened as a PR, confirm the GitHub Actions workflow passes with the same gates.

## Phase 1 Completion Criteria

Phase 1 is complete when:

- A clean checkout can install dependencies with pnpm.
- The app can be built with TypeScript in ESM mode.
- `GET /health` is tested through `app.inject()` and the app closes in teardown.
- Config is loaded and validated in one place.
- Fastify structured logging is configured safely.
- Error responses have a safe Problem Details baseline.
- Docker Compose can start a local Postgres service.
- CI runs format, lint, typecheck, tests, and build.
- README accurately documents setup and commands.
- No product-domain implementation has leaked into Phase 1.
